# 参与贡献

欢迎提 Issue、PR。这是个用爱发电的非商用小项目，没有硬性流程，但以下几点能让协作更顺。

## 提 Issue

- **Bug**：说清系统（macOS/Windows/Linux + 版本）、做了什么、预期 vs 实际、有没有报错截图或日志（`~/.sushiro/sushiro-log.txt`，注意先删掉里面的凭证/token）。
- **功能建议**：说清场景——你想解决什么问题、现在怎么绕过的。越具体越好，"希望更好用"这种没法动手。

## 提 PR

1. Fork → 建分支 → 改 → PR 到 `master`。
2. **本地必须通过**（CI 也会跑，本地先过省时间）：
   ```bash
   gofmt -l .        # 无输出 = 格式 OK
   go vet ./...
   go test ./...
   go build ./...
   ```
3. **大改动先开 Issue 讨论**：动预约/取号/排队/取消/凭证/抓包这些核心逻辑前，先开 Issue 对齐方向，避免白做。
4. **只改该改的**：别顺手重排无关代码、别改格式化风格、别带 IDE 配置文件。

## 代码约定（见 `ARCHITECTURE.md`）

- **纯标准库，零外部依赖**——不要引入第三方 Go 库。
- **分层**：`core`（公共内核，不依赖其他内部包）→ `api`/`proxy`/`platform`/`notify` → `app`（最上层）。别让 `core` 反向依赖上层。
- **平台相关能力走 `platform/`**，业务代码不直接调系统命令。
- **新 Web API 放 `web_*.go`**，新后台生命周期放 `engine_*.go`，避免 `engine.go` 继续膨胀。
- **大 feature 先写 spec**（`specs/NNN-name/`），尤其碰预约/排队/取消/凭证。
- **关键业务位置写中文注释**，解释「做什么 + 为什么」（尤其边界、并发、坑）。

## 风格

- 提交信息用中文或英文都行，说清改了啥。
- 注释跟着代码走，过时的注释比没注释更坑。

## 行为准则

友善、就事论事。这是个人维护的开源项目，没有全职响应，请多点耐心。
