# Feature Spec: Auth Staleness Reminder（令牌过期提醒）

## Metadata

| Field | Value |
|---|---|
| Feature | `auth-staleness-reminder` |
| Version Target | `2.19.0` |
| Owner | `sushiro-overdose` |
| Date | `2026-06-05` |
| Status | **已实现（含顶栏认证状态徽标）** |
| Affected Surfaces | `web 首页横幅 / /api/status / 通知渠道 / 读取类请求错误处理` |

## Problem

认证令牌失效目前**只在用户真正去抢号/取号那一刻**才被发现（`isAuthError` → 删本地配置 → 报"认证已失效"）。看日历、看排队、看我的单据这些读取类请求即使因认证失败而拿不到数据，也没有统一告诉用户"这是认证过期了，去重新捕获"。用户总是**撞墙之后**才知道，体验差、且容易误判成别的故障（如这次误以为是取号 bug 或网络问题）。

## Context / 根因（重要）

根因（用户实测 + 推断，强证据支持）：**工具复用的是手机/PC 微信抓包得到的同一套令牌，工具自身没有独立登录；寿司郎一个账号同一时间只认一个活跃会话。于是"同一套令牌、两个客户端"互斥——谁后用，另一端就掉。**

- 在手机小程序用一次 → 服务器把会话认到手机 → 工具这份令牌随之失效（"老是要重新捕获" = 本 spec 的 Issue 1）。
- 用工具调用一次 → 服务器认到工具 → 手机端会话被顶掉 → 小程序再请求报"**服务器出错**"（用户实测错误文案；已排除手机代理残留——用户未设过手机代理）。

**这不是 bug，是工具复用令牌的先天限制**：工具无法凭空造一个独立会话，因此"电脑用 vs 手机用"在同一套令牌上天然互斥，无法同时都活着。能做的不是"修复互斥"，而是**令牌被顶掉时第一时间提醒用户重新捕获**（本 spec），并在文案里点明"用完工具后手机需重开小程序刷新会话"。

这有两个直接设计后果：

1. **令牌会"频繁"失效**（每次换设备使用都可能失效），所以"过期提醒"价值很高、且应能反复触发。
2. **绝不能用后台定时探活来检测**——任何工具发起的官方请求都可能再次顶掉手机会话，定时探活 = 定时给用户添乱。因此**只能被动检测**。

> 本 spec 只解决"令牌失效时主动、友好地提醒用户重新捕获"。手机端报错（Issue 2）的根因确认与缓解另开议题，不在本 spec 范围。

## Goal

让用户在认证失效时**第一时间、以友好方式**被提醒去重新捕获，而不是撞到隐晦失败才反应过来；并顺带解释"在手机上用过小程序后电脑这边认证就会失效"，校正预期。

## User Stories

| ID | Story | Priority |
|---|---|---|
| US-1 | 作为用户，当任意读取/操作请求因认证失效而失败时，我能在页面顶部看到常驻提示「认证可能已过期，点此重新捕获」。 | P0 |
| US-2 | 作为用户，认证从"有效"变"失效"时，我能通过已配置的通知渠道收到**一次**提醒（不刷屏）。 | P1 |
| US-3 | 作为用户，提示里能解释"在手机用过小程序后电脑认证会失效"，让我明白为什么要重新捕获。 | P1 |
| US-4 | 作为用户，重新捕获成功后，提示和横幅自动消失。 | P0 |

## Scope

### In Scope

- 被动检测：在已有的读取/操作请求路径里，遇到认证失败（`isAuthError` / 401 / 403）时，置一个内存级 `authHealth=stale` 状态（含原因、时间）。
- `/api/status` 增加 `auth_health` 字段（ok / stale / unknown + 最近检测时间 + 原因）。
- Web 首页常驻横幅：stale 时显示「认证可能已过期，点此重新捕获」，点击进入获取认证流程。
- 通知：ok→stale 跃迁时通过 `sendNotification` 推一次；stale 持续期间不重复推。
- 重新捕获/认证成功后，将 `authHealth` 重置为 ok，横幅与提醒消失。

### Out Of Scope

- **不新增任何后台定时探活/轮询**（见 Context：会加剧会话互斥）。
- 不改官方接口、不碰预约/取号/取消 mutation 路径与架构守卫边界。
- 不解决 Issue 2（手机端小程序报错）的根因——另议。
- 不做令牌自动续期（官方无此能力）。

## Behavior

| State/Input | Expected Behavior | User Copy |
|---|---|---|
| 读取/操作请求返回认证失败 | 置 `auth_health=stale`，记录原因+时间 | — |
| `auth_health=stale` | 首页顶部常驻红色横幅 + 点击进入「获取认证」 | `认证可能已过期：在手机上用过寿司郎小程序后，电脑这边的认证就会失效。点此重新捕获。` |
| ok→stale 首次跃迁 | 通过已配置通知渠道推送一次 | `寿司郎 - 认证可能已过期：请在工具里重新获取认证` |
| 重新捕获/认证成功 | `auth_health=ok`，横幅与提醒清除 | — |
| 从未捕获过认证（has_config=false） | 不算 stale，维持现有"未认证"引导，不误报 | — |
| 仅使用只读功能（看排队/数据图） | 不触发认证检测、不打扰 | — |

## Data Rules

| Data | Source | Retention | Can Write Local State | Can Call Official Mutation |
|---|---|---|---|---|
| authHealth(status/reason/at) | 被动来自既有请求结果 | 进程内存，进程退出即失 | No（不落盘） | No |
| 通知发送 | 既有 `sendNotification` | — | No | No |

## Acceptance Criteria

- [ ] 任意读取/操作请求遇认证失败 → `/api/status.auth_health` 变为 stale（含原因、时间）。
- [ ] stale 时首页显示常驻横幅，点击进入获取认证。
- [ ] ok→stale 只推一次通知，stale 持续不重复。
- [ ] 重新捕获成功后 auth_health 回 ok、横幅消失。
- [ ] has_config=false（从没认证过）不误报 stale。
- [ ] 全程无新增后台定时探活/轮询。
- [ ] 不触碰官方 mutation 与架构守卫边界；`go test ./...` 全绿（含架构守卫、前端守卫）。

## Open Decisions

- 横幅是否也在非首页（如「我的单据」「预约未来」）顶部展示，还是只在首页 + 各页加载失败处就近提示。倾向：全局顶部一处常驻即可，避免重复。
- 通知去重窗口：建议"同一 stale 周期只推一次"，重新捕获成功后才允许下次再推。
