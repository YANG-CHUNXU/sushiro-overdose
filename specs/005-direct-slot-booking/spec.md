# Feature Spec: Direct Slot Booking

## Metadata

| Field | Value |
|---|---|
| Feature | `direct-slot-booking` |
| Version Target | `2.18.0` |
| Owner | `sushiro-overdose` |
| Date | `2026-06-05` |
| Affected Surfaces | `web / 预约未来 calendar / booking engine` |

## Problem

「预约未来」日历展示的是**已经放出来、当前可约**的时段，但每个可约时段挂的动作却是「加入狙击」。这混淆了两个概念：

- **狙击（sniper）** 是蹲**还没放出来**的卡片——到放号窗口高频尝试、抢到即停。
- 可约时段是**已经存在的选项**，用户的预期是「点一下直接预约这一个」。

目前没有「直接预约某个具体时段」的入口：唯一的下单方式是首页「自动抢预约」，它按偏好在配置门店里自动挑最优时段，而不是用户点哪个约哪个。结果就是：看到一个能约的时段，却只能「加入狙击」或回首页让程序自己挑。

## Goal

让「预约未来」里**可约**的时段提供「预约这个时段」——直接对这个确切的门店/日期/时段下单；**已满 / 未放出**的时段提供「加入狙击」。两者动作清晰区分，符合用户心智：可约=直接预约，狙击=蹲未放出的卡片。

## User Stories

| ID | Story | Priority |
|---|---|---|
| US-1 | 作为用户，看到一个可约时段，我能点「预约这个时段」直接预约这一个，而不是回首页让程序按偏好挑。 | P0 |
| US-2 | 作为用户，直接预约前会有明确的「会执行操作」确认；成功后能在「我的单据」看到。 | P0 |
| US-3 | 作为用户，这个时段刚好被约满/抢走时，会得到清晰提示并停止，而不是无限尝试。 | P0 |
| US-4 | 作为用户，已满或还没放出的时段，提供「加入狙击」去蹲，而不是直接预约。 | P1 |

## Scope

### In Scope

- booking 引擎新增「钉死单个时段」模式：给定 store/date/start/end 时只认这一个时段、不按偏好扫描。
- `/api/engine/booking` 接受可选 `{store,date,start,end}`；带这些参数即为直接预约该时段，否则维持原「按偏好自动抢」。
- 直接预约是**一次性**：成功→停；满了/被抢/业务错误→明确报错停；认证失效→走既有处理。
- 「预约未来」日历：可约时段动作改为「预约这个时段」；已满/不可约时段提供「加入狙击」。

### Out Of Scope

- 不改狙击本身的逻辑（仍是蹲未放出的卡片、循环到抢到为止）。
- 不改官方下单接口 `CreateReservation` 的签名或调用边界——它仍只出现在 `runBooking` 等已授权入口。
- 不改取消预约/排队号的任何路径。
- 不做批量预约、不做「同时预约多个时段」。

## Behavior

| State/Input | Expected Behavior | User Copy |
|---|---|---|
| 日历时段 = 可约 (AVAILABLE) | 显示「预约这个时段」按钮 | `预约这个时段` |
| 点「预约这个时段」 | 先弹「会执行操作」确认，确认后直接预约该确切时段 | `直接预约这个时段？会向寿司郎提交预约…` |
| 直接预约成功 | 引擎停在成功态，保存到本地，发通知，引导去「我的单据」 | `预约成功！{时段} @ {门店}` |
| 该时段已满/被抢 | 一次性停止并提示，不无限尝试 | `这个时段已经约满或被抢走了` |
| 已有进行中预约 | 走既有 `ErrActiveReservationExists` 处理并停止 | 既有文案 |
| 认证失效 | 走既有认证失效处理 | 既有文案 |
| 日历时段 = 已满/不可约 | 显示「加入狙击」按钮（蹲它放出来） | `加入狙击` |

## Data Rules

| Data | Source | Retention | Can Write Local State | Can Call Official Mutation |
|---|---|---|---|---|
| 选中的时段（store/date/start/end） | Web 请求 | 请求/单次运行生命周期 | No | Yes（仅 `CreateReservation`，在 `runBooking` 内） |
| 预约结果 | `CreateReservation` 返回 | 既有预约记录留存 | Yes（既有下单成功落本地，与自动抢一致） | — |

## Acceptance Criteria

- [ ] `/api/engine/booking` 带 `{store,date,start}` 时只预约该确切时段，不扫描其它时段。
- [ ] 不带参数时维持原「按偏好自动抢」行为不变。
- [ ] 直接预约成功后引擎进入 success 态并保存预约，与自动抢一致。
- [ ] 该时段不可约时一次性停止报错，不进入无限循环。
- [ ] 「预约未来」可约时段显示「预约这个时段」，已满/不可约显示「加入狙击」。
- [ ] 直接预约前有「会执行操作」确认弹窗。
- [ ] `CreateReservation` 仍只出现在 `runBooking`/`runBookingLoop`/`runSniper`/`runSniperLoop`，架构守卫保持通过。
- [ ] 取消预约/排队号路径不受影响。

## Open Decisions

- 直接预约失败（瞬时网络/官方 5xx）是否做 1-2 次快速重试后再停——本期：瞬时错误可短重试，业务性「已满」立即停。
