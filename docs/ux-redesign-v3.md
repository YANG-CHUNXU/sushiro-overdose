# UX 整体改造设计 v3：从「工具」到「吃寿司助手」

> 目标：让一个没有技术背景、只想吃寿司的人，打开应用 10 秒内得到第一个有用的答案，
> 5 分钟内（如果需要）完成凭证获取，并且全程知道"我在哪、下一步干什么"。
> 不新增功能，只重排呈现方式和优先级。

---

## 一、参考产品分析：高门槛效率工具如何做好上手

| 产品 | 它的"前置门槛" | 它怎么化解 | 对我们的启示 |
|------|----------------|------------|--------------|
| Raycast | 要改系统快捷键、给辅助功能权限 | Onboarding 不是"读说明"而是"现在就做一次"：当场设热键、当场跑第一条命令，每完成一步立刻给反馈 | 凭证向导每一步都应该是"做一个动作 → 看到一个状态变化"，而不是读一段四步说明 |
| Arc | 要导入书签、登录、理解新交互模型 | 首次启动是全屏分步向导，**一屏只做一个决定**，可跳过，完成时有庆祝动效；个性文案降低紧张感 | 向导一屏一动作；完成凭证获取值得一个庆祝时刻（这是全产品最难的一步） |
| Keka | 无 | 整个应用就是一个动词（拖进来=压缩），默认值即最佳实践 | 每个页面应该能用一句话说清"这页帮你回答什么问题"；默认值替用户做决定（如提醒点模板） |
| 美团/大众点评 | 无，但功能极多 | 首页不是功能列表而是场景金刚区；**有进行中订单时订单状态卡置顶第一屏**；文案口语化（"还有 3 桌"不是"queue_count=3"） | 首页按"我的状态"组织而非按"功能模块"；有排队号/预约时它就是首页的主角 |

提炼出五条原则，作为本次改造的判定标准：

1. **先给甜头，再要门槛**（value before setup）：用户先看到实时排队这种零门槛价值，再在需要时才被引导获取凭证。
2. **一屏一个决定**：向导和首页的每个视图只要求用户做一件事。
3. **应用替用户记住状态**：下一步永远由应用提示，而不是让用户回忆"我做到哪了"。
4. **状态优先于功能**：首页的组织原则是"你现在的状态 + 你现在能做什么"，不是功能清单。
5. **完成即庆祝，出错给出路**：里程碑（凭证保存、抢到号）有正反馈；错误保留原始技术细节（项目既定原则），但旁边永远有一个可点的出路按钮。

## 三、核心模型：三态首页

整个改造围绕一个用户状态机展开。首页（p-da）根据状态渲染三种形态，这是优先级最高的改动（功能联动 > 新人引导 > 单页细节）：

```
┌─────────────┐  获取凭证成功   ┌─────────────┐  取到号/约到位  ┌─────────────┐
│  游客态      │ ─────────────▶ │  就绪态      │ ─────────────▶ │  进行中态    │
│ (无凭证)     │                │ (有凭证)     │ ◀───────────── │ (有活动单据) │
└─────────────┘                └─────────────┘   单据完结/取消  └─────────────┘
```

判定数据已全部存在：`hc`（has_config）、`ah`（auth_health）、我的单据接口。**不需要新后端能力。**

### 3.1 游客态首页（首次打开 / 无凭证）

回答："这个应用能帮我干嘛？我现在就能用什么？"

```
┌────────────────────────────────────────────────────────────┐
│ 🍣 SUSHIRO Overdose            [只读模式] v2.23             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│   想吃寿司郎？先看看现在排多久 👀                            │
│   选一家你常去的店，马上看到实时排队——不用登录、不用凭证。     │
│                                                            │
│   ┌──────────────────────────────┐                         │
│   │ 🔍 搜索门店（如：国贸 / 太阳宫） │  [查排队]              │
│   └──────────────────────────────┘                         │
│                                                            │
│ ── 你现在想做什么 ──────────────────────────────────────────│
│ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│ │只读·直接用    │ │只读·直接用   │ │需要通行证 🎫 │            │
│ │现在想去吃    │ │我已拿到号    │ │约未来/自动取号│            │
│ │看哪家排得少  │ │估几点叫到我  │ │抢周末、远程蹲 │            │
│ └─────────────┘ └─────────────┘ └─────────────┘            │
│                                                            │
│ ▸ 高级与设置（折叠：偏好 / 通知 / 预测准确度 / 日志）          │
└────────────────────────────────────────────────────────────┘
```

要点：

- **hero 主动作改为"搜门店查排队"**——10 秒甜头。替代现在的「开始/获取凭证/高级：重置抓包」三按钮（A1/A2）。「重置抓包」移入设置-高级区。
- 选定的门店写入"常用门店"（已有 selected_stores 偏好），全站默认带入：门店排队、叫号预测、日历都不再要求重选（联动）。
- 任务卡从 4 张减为 3 张：「自动蹲号」并入第三张"约未来"（A8）。卡片文案从功能描述改为收益描述（"看哪家排得少"而非"查看全国门店实时排队、营业状态…"）。
- 需要凭证的卡用 🎫「通行证」隐喻替代"凭证"一词的裸奔（详见 §6 文案表）。
- summary minis（A6）游客态整体不渲染。

### 3.2 就绪态首页（有凭证、无活动单据）

回答："一切就绪，今天怎么吃？"

- hero 显示凭证健康 + 常用门店当前排队摘要（"国贸北店现在约 23 组 · 预计 40 分钟"），主按钮「现在取号」/「查可约时段」。
- 保留任务卡，但顺序按使用频率重排：取号 > 看排队 > 约未来。
- 「准备清单」卡（替代现在的"第一次使用"四步 track）：通行证 ✅ / 通知渠道 ⚠️未配置 / 预测数据 ⚠️建议开启——即 design-system 已定义的 RequirementStrip，三项前置条件集中一处，各带一个修复按钮。配齐后整卡消失。

### 3.3 进行中态首页（有排队号或预约）——本次最重要的新视图

回答："我的号到哪了？我该几点出发？"（美团订单卡模式，A3）

```
┌────────────────────────────────────────────────────────────┐
│ 🎫 你正在排：国贸北店 · B1024                                │
│                                                            │
│    现在叫到 B0987 · 你前面还有约 37 组                       │
│    ⏱ 预计 18:40 叫到 · 建议 18:20 前出发                    │
│    ▂▃▅▆▇ 叫号速度曲线（最近 1 小时）                         │
│                                                            │
│    [🔔 提醒我（已设 3 个提醒点）] [查看详情] [⋯ 取消排队号]   │
├────────────────────────────────────────────────────────────┤
│ （下方再放常规任务卡，弱化）                                  │
└────────────────────────────────────────────────────────────┘
```

- 数据全部来自既有接口（我的单据 + 叫号预测 + 提醒状态），是纯前端联动。
- 取消排队号收进 ⋯ 菜单并二次确认（已有 confirmDialog danger 样式）。
- 多张单据时纵向堆叠，最近的在上。

## 四、首次使用流程（FTUE）重设计

### 4.1 流程总览

```
首次打开
  └─▶ 欢迎屏（一次性，可跳过）
        "选一家你常去的寿司郎" → 搜索/选择 → 立即跳到该店实时排队页
              │
              ▼
        （甜头已给。此后不再打扰，直到用户碰到需要凭证的功能）
              │
       用户点"约未来/取号" 且无凭证
              ▼
        通行证向导（状态机，可中断恢复）
```

关键变化：现在的 `openFirstUseWizard`（四卡选场景弹窗）**降级为欢迎屏的一部分**，且把"选常用门店"提到第一步——选完门店无论走哪个场景都有默认上下文，这是最大的联动收益。

### 4.2 通行证向导：从说明书到状态机

现状（A5）：每步一大段 `<ol>` 文字。改为五步状态机，**一屏一动作，底部常驻步骤条**：

```
●━━━━○━━━━○━━━━○━━━━○
选设备   抓一次   传到电脑  粘贴    验证

┌─ 第 2 步 / 共 5 步：在手机上"抓一次" ─────────────────────┐
│                                                          │
│  打开 Stream（已装好✓），点「开始抓包」，然后：              │
│                                                          │
│   1️⃣ 微信里打开寿司郎小程序                                │
│   2️⃣ 随便点一家门店  ← 这一下会产生我们要的"查询请求"        │
│   3️⃣ 再点一次「我的预约」 ← 这一下产生"预约请求"             │
│                                                          │
│  💡 为什么要点两次？两类请求各含一半通行证信息。              │
│                                                          │
│             [我点完了，下一步 →]    [← 上一步]              │
└──────────────────────────────────────────────────────────┘
```

各步设计：

| 步 | 屏幕上唯一的动作 | 进度反馈 |
|----|------------------|----------|
| 1 选设备 | 点 iPhone / 安卓 /（macOS 额外给"PC 微信自动抓"入口） | — |
| 2 抓一次 | 跟着 3 个编号动作操作，点"我点完了" | — |
| 3 传到电脑 | 文件传输助手图示（手机→💬→电脑），点"内容已在剪贴板" | — |
| 4 粘贴解析 | 一个大文本框 +「解析」 | **8 字段实时 checklist**（复用 `need[]` 和 `rG()` 的逻辑搬进向导）：`✅ X-App-Code ✅ 查询凭证 ⬜ 预约凭证 …`；缺哪类就提示"回到第 2 步再点一次『我的预约』"，并保留已捕获字段（后端 `/api/auth/import` 已支持增量合并） |
| 5 验证 | 自动调用"测试基础接口"，转圈→结果 | 成功：🍣 庆祝动效 + "通行证已生效，去约一个？"；失败：原始错误全文 + 「重试」「回到第 4 步」 |

可中断恢复：向导步骤与已捕获字段数存 localStorage（`sushiro_wizard_state`），中途关闭后，就绪清单卡显示"通行证获取到一半（5/8 项）· 继续"。后端字段状态以 `/api/auth/import` 返回的 `missing[]` 为准，本地只存"走到第几步"。

自动代理抓（同 Wi-Fi 路径）同样拆为子步骤（装证书→信任→设代理→点小程序），每步等待对应的服务端信号（已有 `/api/mobile-auth` 轮询：CA 下载、首个请求、字段捕获均可观测），**用信号驱动步骤自动前进**，而不是一段 8 步文字让用户自己核对。

### 4.3 凭证过期的再引导

stale 时的 authBanner 保留，但文案改为人话 + 原始错误折叠展示：

> 🎫 通行证可能失效了（寿司郎会定期回收，或被手机端登录顶掉）。
> [重新获取（约 3 分钟）]  ▸ 技术细节：`E010/error.server`

符合"错误原文保留"原则：原文不删，折叠陈列。

## 五、信息架构重构

### 5.1 导航（NAV_GROUPS 改动）

```
现在：首页 | 现在去吃 | 我有号码 | 预约未来 | 自动蹲号 | 我的单据 | 设置
改为：首页 | 现在去吃 | 我有号码 | 约未来   | 我的单据 | 设置
```

- 「自动蹲号」(sn) 移入 book 组：`pages:[['ca','可约日历'],['sn','自动蹲号'],['in','历史洞察']]`。狙击的自然入口本来就是日历里"已满→加入狙击"（`snFromSlot` 已实现），一级导航反而制造了"我该从哪进"的困惑。
- 「我有号码」三个子页标签已经很口语（取号→几点吃），保留。
- 设置组的「预测准确度」「日志」保留为子页，但页内重组（见 5.2）。

### 5.2 设置页分层（A7）

单页滚动改为三段折叠分区（移动端友好，避免 tab 状态丢失）：

1. **基本**（默认展开）：人数/儿童/桌型、常用门店与优先级、通知渠道。通知渠道每项配置后给「发送测试通知」即时反馈。
2. **通行证与数据**：凭证状态卡（健康度/重新获取/重置）、云端服务地址、预测数据开关。"凭证内容粘贴框""手动粘贴 UA"从设置页移除，统一走向导（设置页只留"重新获取"入口）。
3. **高级**（默认折叠，标注"一般用不到"）：重置抓包、调试模式、诊断包、日期/时段策略微调。

### 5.3 全局状态语言统一

- 右上角 authPill 升级为**健康胶囊**：综合通行证/通知/预测数据三项，文案取最差项（"通行证已就绪""通知未配置""通行证可能失效"），点击展开 RequirementStrip 弹层。这样三个前置条件在任何页面都有同一个查看入口，各功能页不再各自为政地提示。
- 三个安全标签全站强制使用且只用这三个词：`只读·直接用` / `需要通行证` / `会执行操作·需确认`。

## 六、视觉与品牌

### 6.1 保留与延伸

现有 token（--red #B81C22、纸色系、绿/黄语义色）成熟，不动。延伸三点：

1. **吉祥物**：一个简笔 SVG 寿司角色（建议：一贯三文鱼握寿司"小贯"，圆眼、有手）。固定用于四个场景，不滥用：
   - 空状态（歪头："这几家店现在没有放出时段，晚点再来看看？"）
   - 向导完成（举手庆祝 + 少量 🍣 confetti，一次性动画 <1.5s）
   - 等待中（进行中态首页，闭眼打盹 = 离叫号还远；睁眼 = 快到了）
   - 404/错误页脚（鞠躬道歉，旁边仍是原始错误全文）
2. **字阶与留白**：hero 标题升到 28-32px，正文行高 1.6；卡片圆角从现状统一到 14-16px；新手可见区域的信息密度下调（每卡一句话+一个按钮），高级区维持现在的紧凑密度。
3. **动效原则**：只在"状态变化"时动（步骤完成打勾、号码前进、捕获字段点亮），装饰性动画为零。全部 CSS transition，纯标准库约束下无需引入依赖。

### 6.2 文案对照表（主路径用左列，高级区/日志保留右列）

| 用户词（主路径） | 技术词（高级区/错误原文保留） |
|------------------|------------------------------|
| 通行证 | 凭证 / Authorization / X-App-Code |
| 拿一次通行证（约 3 分钟） | 抓包 / 捕获认证参数 |
| 寿司郎会定期回收通行证 | token 过期 / 401/403 / E010 |
| 电脑帮手机看一眼网络 | MITM 代理 / CA 证书（向导内首次出现时给一句白话解释，词本身不避讳——装证书步骤绕不开） |
| 预测数据 | 采样 / sampling |
| 你前面还有约 N 组 | groupQueuesCount（接口语义：wait=分钟，groupQueues=当前叫号） |

规则：错误信息永不翻译转写（既定原则）；技术词在向导里**第一次出现时**用一行"💡 为什么"解释，之后不再解释。

## 七、落地路线（按既定优先级：功能联动 > 新人引导 > 单页细节）

### P0 · 三态首页与活动单据置顶（联动核心）
