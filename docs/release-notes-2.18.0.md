# Sushiro Overdose 2.18.0 Release Notes

2.18.0 修正「预约未来」里的一个概念混淆：日历展示的是**已经放出来、当前可约**的时段，现在可以**直接预约这一个**，而不是只能「加入狙击」或回首页让程序按偏好挑。狙击回归它本来的含义——蹲**还没放出来**的卡片。

## 主要变化

- 「预约未来」可约时段新增「预约这个时段」：直接对这个确切的门店/日期/时段下单，预约前有「会执行操作」确认，成功后在「我的单据」查看。
- 已满 / 还没放出的时段提供「加入狙击」去蹲它放出来。
- 直接预约是**一次性**：成功即停；这个时段已满/被抢则明确提示并停止，不会无限尝试；认证失效走既有处理。
- 首页第 4 步等文案校正：可约直接预约、未放出再狙击。
- 数据图「我有号码」去掉跨店推荐（原「当前排队参考 / 当前更好去的门店」）：工具不知道你的地理位置，排队短不代表离你近，这种推荐没有意义。数据图现在只专注于你选的那家店「几点叫到多少号」和你的号几点叫到。（撤回 spec 003 的 US-4 门店对比）

## 实现要点

- booking 引擎新增「钉死单个时段」模式：给定 store/date/start/end 时只认这一个时段，不按偏好扫描。
- `/api/engine/booking` 接受可选 `{store,date,start,end}`；带参数即直接预约该时段，空 body 维持原「按偏好自动抢」，向后兼容。
- 官方下单接口 `CreateReservation` 仍只出现在 `runBooking` 等已授权入口内（pinned 逻辑写在 `runBooking` 方法体里），架构守卫不变、保持通过；官方 mutation 边界没有扩大。

## 取号请求贴近官方（防御性）

- 所有官方请求补上微信 WebView 的 `Xweb_xhr` 头，更贴近小程序真实请求。
- 远程取号人数/桌型兜底：未配置预约偏好时不再可能拿"0 人"提交（默认 2 成人、桌位），避免这类输入触发官方 E010。
- 注：取号成败最终取决于官方在该门店/时段是否放行；接口抓包确认本工具的请求字段与官方一致，E010 多为官方当时不可提交。

## 不包含的变化

- 不改狙击本身逻辑（仍是蹲未放出卡片、循环到抢到为止）。
- 不改取消预约 / 取消排队号的任何路径。
- 不做批量或多时段同时预约。

## 验收点

- 「预约未来」可约时段显示「预约这个时段」，已满/不可约显示「加入狙击」。
- 点「预约这个时段」有确认弹窗；确认后只预约这一个确切时段。
- 这个时段被约满/抢走时，一次性停止并提示，不无限尝试。
- 首页「自动抢预约」（空 body）行为不变。
- `go test ./...` 全绿，含架构守卫与前端契约守卫。
