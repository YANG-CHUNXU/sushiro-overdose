# UX command center plan: v3.1.5

## 背景

这次从代码逻辑和实际使用路径看，主要问题不是功能缺少，而是用户打开首页后要自己判断：

- 哪些功能可以直接看，不需要通行证。
- 哪些功能需要通行证，但还不会提交到寿司郎。
- 哪些按钮会真的启动预约、蹲号、取号等动作。
- 出错时应该先处理通行证、代理、证书、网络还是通知。

README 不能代表当前产品状态，所以方案直接以 `internal/app/web_static.go` 的现有页面和状态数据为准。

## 目标

1. 首页第一屏回答“今天该走哪条路”，而不是只展示功能入口。
2. 明确区分“只读 / 通行证 / 会执行”，降低误点重操作的心理成本。
3. 诊断页第一眼给出唯一优先建议，芯片和日志留作支撑信息。
4. 不新增后端能力，不打断老用户现有流程。

## 改动

### 首页指挥台

在首页 hero 下方新增 `journeyPanel`。它会根据当前状态生成下一步建议：

- 无通行证：先选门店看排队，想抢预约再拿通行证。
- 通行证可能失效：建议先重新获取，排队查询仍可用。
- 有未完成单据：优先查看我的单据或叫号预测，避免重复操作。
- 正在运行任务：提示查看日志或先停止当前任务。
- 已就绪：引导到可约日历或自动抢。

面板内固定展示三类能力：

- 只读：排队、预测、叫号估算。
- 通行证：抢预约、远程取号、读取我的单据前置条件。
- 会执行：会向寿司郎发起动作的任务，并继续保留确认弹窗。

### 诊断优先建议

在“安全与维护”里的诊断芯片上方新增 `diagNext`，把分散的红黄绿状态收敛成一个优先动作：

1. 代理残留。
2. 通行证不完整。
3. 证书未信任。
4. 代理链路异常。
5. 寿司郎网络不可达。
6. 未选择门店。
7. 未配置通知。
8. 引擎运行错误。
9. 全部正常。

### 回归保护

新增 `TestEmbeddedUXCommandCenterAnchors`，固定检查首页指挥台、诊断建议、关键文案和渲染函数，避免以后整理内嵌页面时把体验路标误删。

## 验证计划

- `go test ./internal/app`：包含新增体验锚点测试和内嵌 JS 语法检查。
- `go test ./...`：确认全项目测试通过。
- `go vet ./...`：确认静态检查通过。
- 本地浏览器冒烟：打开首页和设置诊断区，确认 `journeyPanel` 与 `diagNext` 渲染、无脚本错误、移动宽度不挤压。
