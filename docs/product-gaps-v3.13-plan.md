# 产品 / UX 缺陷盘点与改造方案（基于 v3.13.4 代码）

> 这份是在通读 `internal/app/` 现有实现后写的，重点是「现有机制还差哪一步」，
> 而不是凭空提需求。每条都标了现状、缺陷、改造、落地文件、优先级。

四个方向，按投入产出排序：

1. 通行证体验（最高优先级）
2. 预测精度的自证闭环
3. 移动端触达
4. 设置页认知负荷收敛

---

## 1. 通行证体验：从「被动报错」到「主动续期」

### 现状（已有的）
- `auth_health.go`：三态状态机 `unknown/ok/stale`，**被动**检测——只看用户已触发的凭证请求结果，不后台轮询（避免把手机会话顶掉）。`ok→stale` 跃迁时推一次通知，去重良好。
- `auth_probe.go`：`/api/auth/probe` 主动探测凭证各接口可用性，用于诊断。
- 设置页有「寿司郎通行证」折叠卡 + 首启向导 `openFirstUseWizard` + 捕获引导 `mobile_auth_capture.go`。

### 缺陷
1. **stale 通知是「死胡同」**：失效通知文案是「请在工具里重置认证并重新获取凭证」——把用户踢回一个多步手动流程（重置→抓包/PC 捕获→验证）。通知本身不可点、不能直接拉起向导。
2. **没有凭证「年龄/寿命」概念**：只有 ok/stale 二元，没有「已用 N 天、同类历史平均 M 天后失效、建议本周内择机重抓」这种**预期管理**。用户永远是被失败打断，而不是被提前提醒。
3. **失效原因对用户不可辨**：stale 的 reason 只有「401/403」或「E010」，用户分不清是「自然过期」还是「手机刚开过小程序顶掉」——而后者其实下次别在手机开小程序就能避免。
4. **重新捕获没有「续期」捷径**：每次都走完整向导，没有「沿用上次的捕获方式（PC 微信 / 手机抓包）一键重跑」。

### 改造
- **A. 通知可操作化**：`markAuthStale` 推送的通知带 deep-link（`http://127.0.0.1:<port>/?recapture=1`），点开直接落到通行证向导并预选上次方式。Bark/Telegram 都支持带 URL，飞书卡片可带按钮。
  - 落地：`auth_health.go`（通知体加 URL）、`web_static.go`（启动读 `?recapture=` 自动 `startAuth()`）。
- **B. 凭证年龄追踪**：捕获/导入成功时记录 `captured_at`；在通行证卡和首页常驻一条「凭证已用 N 天」。失效时记录本次寿命，滚动出该用户的中位寿命，据此在「接近历史寿命」时**提前**推一条柔性提醒（区别于 stale 硬失败）。
  - 落地：`config.json` 增 `captured_at`/`last_stale_at`；`auth_health.go` 增 `authHealthSoftWarn`；前端通行证卡展示。
- **C. 失效原因归因**：stale 时若能拿到「最近一次手机端活跃」线索就提示「疑似手机端顶掉」，否则按「自然过期」。即使做不到精确归因，也把两种可能写进文案 + 给出「下次怎么避免」一句。
  - 落地：`auth_health.go` reason 文案分支。
- **D. 一键续期**：向导记住上次成功方式（`capture_method: pc_wechat | mobile_proxy`），失效后默认直接进该方式，跳过选择步。
  - 落地：`mobile_auth_capture.go` / `web_cloud_auth.go` 持久化方式；向导入口读取。

### 为什么这是第一优先级
登录免费的「号预测」体验已经很完整；通行证是所有**预约系**功能价值的唯一闸门，而它现在三重崖（取得难、信任难、失效频繁）。把失效→续期的环路磨到「点一下通知就好」，比加任何新功能都值。

## 2. 预测精度：从「先验置信度」到「回测校准」

### 现状（已有的）
- `queue_advisor.go`：实时快照 + 本机采样 + 历史趋势合成等待区间，输出 `confidence`、`source/source_label`、`wait_minutes_range`。
- `queue_advisor_accuracy_test.go` / `_dynamic_test.go`：算法层有收敛性、噪声鲁棒性、区间覆盖率等测试（合成数据，±25% 边界）。
- `buildQueuePressureCurve`：本机点数不足时用线上 Turso 基准兜底，来源标注清晰。

### 缺陷
1. **`confidence` 是先验启发式，不是实测**：来自样本数 + 变异系数（cv），代表「数据多不多/稳不稳」，**不代表历史上预得准不准**。用户看到「高置信」但仍可能被某店系统性低估。
2. **没有「预测 vs 实际」对账**：`history.jsonl` 有数据，但没有在「用户当天目标号真正被叫到时」回填一条 `predicted_called_at` vs `actual_called_at`，也就拿不到**每店实测误差**。
3. **区间宽度不按店校准**：所有店共用同一套区间逻辑，没用「这家店历史误差大 → 区间放宽」来自适应。
4. **的中率对用户不可见**：无法说出「这家店我们历史预测平均 ±N 分钟」这种能直接建立信任的话。

### 改造（数据已经在手，主要是利用）
- **A. 回测记录**：每次给出 ETA 时落一条 `{store, target_no, predicted_range, predicted_mid, ts}`；当后续快照显示该号已被叫到，回填 `actual_called_at`，算出该次误差。
  - 落地：新增 `queue_advisor_backtest.go` + 一个 jsonl（如 `eta_backtest.jsonl`）；在快照入口（`queue_live.go` / `sampling.go` 写入处）做回填。
- **B. 每店实测误差面板**：在「预测准确度」折叠卡里展示每店 MAE / P50 / 偏差方向（系统性偏早还是偏晚）。
  - 落地：`insights.go` 旁加聚合；前端 `qdAccuracy` 区。
- **C. 用实测误差校准区间**：把 `confidence` 升级为「先验稳定度 × 实测历史误差」两路融合；历史误差大的店自动放宽 `wait_minutes_range`，并修正系统性偏差（如某店总偏早 12 分就整体右移）。
  - 落地：`queue_advisor.go` 区间生成处接入 backtest 聚合。
- **D. 信任话术**：答案卡补一句「这家店我们最近 N 次预测平均偏差 ±M 分钟」，没数据时老实说「样本不足，按通用模型估算」。
  - 落地：`web_static.go` `qdAnswer` 渲染。

### 注意
回测要排除「用户没按建议出发」造成的污染——只统计「号确实被叫到」的客观时刻与当初预测的差，不掺杂用户行为。

## 3. 移动端触达：核心场景在手机，UI 却锁在桌面

### 现状
- `web.go`：**只绑 `127.0.0.1`**，并有 Host 校验防 DNS rebinding（`isAllowedWebHost` 仅认 127.0.0.1/localhost/::1）——这是刻意的安全设计。
- 外出时唯一触点是通知（飞书/Telegram/Bark/Server酱），支持分阶段「该出发了」提醒。

### 缺陷
「拿号后找地方坐着等」是**移动**时刻，但富 UI 锁在 PC。用户在外面只能收到单向通知，无法主动问「现在到几号了 / 我还要多久 / 要不要现在走」。

### 改造（三选一，按侵入性递增；推荐 A→B）
- **A.（推荐，零安全代价）通知升级为「自更新进度卡」**：把分阶段提醒做成内容更足的卡片——当前叫到、剩余桌数、实时速度、最新出发建议、本次预测误差。飞书用 interactive card，Telegram 用 `editMessageText` **原地更新同一条消息**（不刷屏），Bark 用分级提醒。用户不用交互就能在锁屏看到最新进度。
  - 落地：`internal/notify/` 各 provider 增富内容；`queue_alerts.go` 维护「同一会话的 message_id」做原地更新。
- **B. Telegram 双向 bot**：用户在 Telegram 回「进度」/「还要多久」，bot 拉本机最新 advisor 结果回。利用 Telegram long-poll，本机出站连接，**不需要对外开端口**，安全模型不变。
  - 落地：`internal/notify/telegram_bot.go` + 复用 `queue_advisor` 查询。
- **C.（谨慎）同 WiFi 手机访问**：可选开关，绑 LAN IP + 一次性 token + 二维码配对。和现有安全立场冲突最大，**默认关、要用户显式开 + 强警示**。
  - 落地：`web.go` 绑定可选项 + token 中间件；不建议作为默认路径。

### 建议
先做 A（投入小、不动安全模型、直接提升核心场景体验），B 作为进阶。C 除非有明确需求否则不碰。

## 4. 设置页认知负荷：9 张折叠卡拍平排列

### 现状
设置页有 9 个 `setting-fold`：本机持续采集、通行证、GitHub 线上基准、通知渠道、预测准确度、历史洞察、诊断与维护、运行日志、MCP 助手。已有 `simple-mode`/`advanced-mode` 切换（simple 隐藏 `advanced-only`）。技术词（Turso）已用「数据库地址/只读密钥」等做了产品化包装。

### 缺陷
1. **9 张卡平铺、无层级**：日常项（通知、通行证）和重运维项（采集、Turso 基准、MCP、日志）混在一个长列表里，找东西靠滚。
2. **进阶基础设施仍在膨胀**：MCP / Turso / Worker / 采样守护进程不断往里加，advanced 模式正在变成「什么都往里塞」。
3. **缺一个状态总览**：用户进设置常是「想确认某项是否正常」，但现在要逐个展开看。

### 改造
- **A. 分组分区**（在现有 fold 外包一层小标题）：
  - 「日常」：通知渠道、通行证
  - 「让预测更准」：本机采集、预测准确度、历史洞察、线上基准
  - 「进阶 / 集成」：MCP 助手
  - 「排障」：诊断与维护、运行日志
  - 落地：`web_static.go` 设置页加分区标题，不改 fold 本身。
- **B. 顶部状态总览条**：一行 chip 汇总——通行证(ok/stale/未配置)、通知(已配 N 渠道)、采集(开/关)、线上基准(连通/未配)。点 chip 跳到对应卡。复用已有 `/api/status` 字段。
  - 落地：`web_static.go` 设置页顶部；数据来自现有 status 轮询。
- **C. simple 模式只留「日常」+ 状态总览**，其余整组归 advanced。让新用户的设置页只剩通知和通行证两件事。

### 优先级
低于前三项，但 A+B 改动小、纯前端、风险低，可作为顺手清理。

## 一句话总结

号预测（免登录）是这个产品已经验证的护城河；**真正卡价值兑现的是通行证环路**。所以排序是：
先让通行证「失效→续期」无痛（#1）→ 用已有数据把预测做到可自证可信（#2）→ 把核心移动场景从单向通知升级为自更新进度（#3）→ 顺手收敛设置页（#4）。
